- `feat/feature-name` — для новых функций.
- `fix/bug-name` — для исправления ошибок.
- `docs/readme-update` — для документации.


    main[(main)] -->|создание ветки| feat(feat/login)
    feat -->|коммит 1| feat
    feat -->|коммит 2| feat
    feat -->|Pull Request| main


1. **Обновите локальный `main`**:  
    `git checkout main && git pull origin main`
2. **Создайте целевую ветку**:  
    `git checkout -b feat/user-auth`


- **Игры с индексацией**: Используйте `git add -p`, чтобы добавлять в индекс конкретные строки кода, а не весь файл целиком.
- сообщения по шаблону: `тип(контекст): описание`.
    - _Правильно_: `feat(auth): add JWT token validation`
    - _Правильно_: `fix(db): resolve memory leak in connection pool`
    - _Неправильно_: `fix errors`, `update code`, `done`.

🔀 4. Слияние (Merge) через Pull Request
1. **Отправьте ветку в удаленный репозиторий**:  
    `git push origin feat/user-auth`
2. **Создайте Pull Request (PR)** на GitHub/GitLab.
3. **Имитируйте Code Review**:
    - Откройте вкладку _Files Changed_.
    - Внимательно просмотрите свой код глазами чужого человека.
    - Найдите и удалите забытые `console.log`, `print` или временные комментарии.
4. **Слияние**: Нажмите **Squash and Merge**. Это объединит ваши мелкие локальные коммиты в один красивый коммит внутри ветки `main`.
5. **Удаление**: Удалите ветку `feat/user-auth` на GitHub и локально (`git branch -d feat/user-auth`).

Каждый раз при сохранении данных проекта (коммите) система фиксирует состояние файла (делает снимок) и создает ссылку на этот снимок. Последующие изменения отражаются через ссылки на более ранние версии файла. Нет необходимости снова сохранять файл целиком. К тому же, основываясь на контрольных hash-суммах, система снимков обеспечивает целостность всей истории изменений. На практике это означает, что невозможно (либо крайне трудно) полностью удалить данные из рабочего каталога и утратить к ним любой доступ. В большинстве случаев данные можно восстановить из ранней версии проекта.  

У Git есть рабочий каталог, где хранятся метаданные и локальная база рабочего проекта. Именно эта часть копируется, когда вы **клонируете** проект (репозиторий) с сервера.  


      `> git config --global core.editor emacs`


Правила и периодичность обновления могут быть почти любыми, но хорошим тоном обычно считается сохранять рабочую (или промежуточно завершенную) версию. Важное требование для команд разработчиков — возможность сборки проекта, иначе другие участники команды будут вынуждены тратить время на борьбу с ошибками компиляции.
